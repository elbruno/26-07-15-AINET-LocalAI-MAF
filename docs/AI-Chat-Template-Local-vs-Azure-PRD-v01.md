## 1. Summary

Build **one repository with two folders**, each a working instance of Microsoft's **AI Chat Web App** template (a Blazor RAG chat app that chats over your PDFs):

1. **`01-aichatweb-azure`** — the template **unchanged**, using **Azure OpenAI** for both chat and embeddings, with a local on-disk vector store. This is the "cloud baseline."
2. **`02-aichatweb-local`** — the **same app, made fully on-device**: the chat LLM is served by **Foundry Local**, and the embeddings are generated by **`ElBruno.LocalEmbeddings`**. The vector store stays local. **Zero cloud calls, no API keys.**

The whole point of the pairing is to show that **only two seams change** — `IChatClient` (chat) and `IEmbeddingGenerator` (embeddings). The Blazor UI, the RAG pipeline, the data ingestion, and the vector store are identical.

## 5. Folder 1 — `01-aichatweb-azure` (cloud baseline)

### 5.1 Generate
From the repo root:
```bash
dotnet new aichatweb --Framework net10.0 --provider azureopenai --vector-store local --output 01-aichatweb-azure
```

### 5.2 Configure (keyless Azure OpenAI)
- Set user-secrets in the project (do **not** commit secrets):
  ```bash
  cd 01-aichatweb-azure
  dotnet user-secrets set "AzureOpenAi:Endpoint" "https://<your-resource>.openai.azure.com/"
  ```
- Authentication is **keyless** via `DefaultAzureCredential` (Azure CLI login). No API key in source.
- Model names (as generated by the template) — keep the defaults unless your deployments differ:
  - Chat: `gpt-4o-mini`
  - Embeddings: `text-embedding-3-small`

### 5.3 The two seams (reference — leave as generated)
`Program.cs` registers both via the Azure OpenAI client, e.g.:
```csharp
var azureClient = new AzureOpenAIClient(
    new Uri(builder.Configuration["AzureOpenAi:Endpoint"]!),
    new DefaultAzureCredential());

builder.Services.AddChatClient(
    azureClient.GetChatClient("gpt-4o-mini").AsIChatClient());

builder.Services.AddEmbeddingGenerator(
    azureClient.GetEmbeddingClient("text-embedding-3-small").AsIEmbeddingGenerator());
```
> Copilot: use the exact registration the template emits for `--provider azureopenai`; do not hand-rewrite it. This block is shown only so the *diff* against Folder 2 is obvious.

### 5.4 Run
```bash
dotnet run
```
**Acceptance:** the app starts, ingests the sample PDFs, and answers a question (e.g. *"What are some essential tools in the survival kit?"*) **with citations**.

## 6. Folder 2 — `02-aichatweb-local` (Foundry Local + ElBruno.LocalEmbeddings)

The strategy: **generate the same template, then replace only the two seams.** Everything else stays identical to Folder 1.

### 6.1 Generate the base
```bash
dotnet new aichatweb --Framework net10.0 --provider ollama --vector-store local --output 02-aichatweb-local
```
> Rationale: the template has no built-in "Foundry Local" provider, so start from a **local-oriented** scaffold (`ollama`) — it already assumes a local, keyless, OpenAI-compatible chat endpoint and a local vector store, which minimizes the edits. Copilot will then repoint the chat client to Foundry Local and replace the embedding generator.

### 6.2 Add the local packages
```bash
cd 02-aichatweb-local
dotnet add package ElBruno.LocalEmbeddings
# Foundry Local SDK (on-device model manager + OpenAI-compatible endpoint):
dotnet add package Microsoft.AI.Foundry.Local
```
(If the project references `Microsoft.Extensions.AI.Ollama`, it can be removed once the chat client is repointed.)

### 6.3 Seam #1 — chat → Foundry Local
Replace the chat-client registration in `Program.cs` so `IChatClient` points at the Foundry Local OpenAI-compatible endpoint. Use the Foundry Local SDK to start/resolve the model and its local endpoint + key:
```csharp
using Microsoft.AI.Foundry.Local;
using OpenAI;
using System.ClientModel;

// Start (or attach to) a local model and get its OpenAI-compatible endpoint.
var alias = "phi-4-mini";
var foundry = await FoundryLocalManager.StartModelAsync(alias);
var info = await foundry.GetModelInfoAsync(alias);

var openAi = new OpenAIClient(
    new ApiKeyCredential(foundry.ApiKey),          // local, non-secret
    new OpenAIClientOptions { Endpoint = foundry.Endpoint });   // e.g. http://localhost:<port>/v1

builder.Services.AddChatClient(
    openAi.GetChatClient(info.ModelId).AsIChatClient());
```
> Alternative (no local REST server): use Bruno's in-process adapter **`ElBruno.MAF.FoundryLocal.Adapter`** — `FoundryLocalChatClientAdapter` exposes Foundry Local directly as an `IChatClient`, and the same client can back a Microsoft Agent Framework agent.
> Copilot: adapt names to the installed SDK surface if they differ; the **contract** is "register an `IChatClient` backed by Foundry Local." Keep the rest of the chat pipeline untouched.

### 6.4 Seam #2 — embeddings → ElBruno.LocalEmbeddings
Replace the embedding-generator registration so `IEmbeddingGenerator` is the local ONNX generator (default model `all-MiniLM-L6-v2`, **384 dims**):
```csharp
using ElBruno.LocalEmbeddings;

// LocalEmbeddingGenerator implements IEmbeddingGenerator<string, Embedding<float>>.
var localEmbeddings = await LocalEmbeddingGenerator.CreateAsync();
builder.Services.AddEmbeddingGenerator(localEmbeddings);
```
> Everything downstream (ingestion, vector store, semantic search) already programs against `IEmbeddingGenerator`, so no other code changes are required.

### 6.5 Keep the vector store local (unchanged)
The template's local on-disk `JsonVectorStore` is reused as-is. **Important:** the embedding dimension changes from 1536 (Azure `text-embedding-3-small`) to **384** (MiniLM), so the ingestion cache/vector store from any prior run must be cleared before first run:
- Delete the template's ingestion cache / on-disk vector store directory (commonly under `bin/`/`obj/` or a configured `vector-store`/`ingestioncache` folder) so ingestion **re-embeds** the PDFs at 384 dims on startup.

### 6.6 Run
```bash
# 1) make sure the model is available locally (once):
foundry model run phi-4-mini
# 2) run the app (re-ingests PDFs with local embeddings on first start):
dotnet run
```
**Acceptance:** the app starts with **no Azure sign-in and no keys**, ingests the sample PDFs using local embeddings, answers a question **with citations**, and — verified via a network monitor — makes **zero outbound AI calls**.

## 11. Gotchas (call these out in the README)

- **.NET 10 required** for `ElBruno.LocalEmbeddings` — generate both folders with `--Framework net10.0`.
- **Dimension mismatch (1536 → 384):** clear the ingestion cache / vector store before the first local run, or ingestion will fail to match an existing index.
- **Foundry Local must be running** and the chat model pulled (`foundry model run phi-4-mini`) before `dotnet run` in Folder 2.
- **Model choice:** `phi-4-mini` is a good small default; any Foundry Local chat model works — set the alias in one place in `Program.cs`.
- Optional upgrades for Folder 2: `ElBruno.LocalEmbeddings.Npu` (DirectML/NPU acceleration) and the `.Harrier` model (640-dim, multilingual) — noted but out of scope for the baseline.
